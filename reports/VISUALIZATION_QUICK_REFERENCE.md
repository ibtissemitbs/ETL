## 📖 Captions Prêtes pour Publication

### Figure 1 - Pipeline Timing
```
Figure 1: Decomposition of ETL pipeline execution time by stage. 
The complete pipeline executes in 5.0 seconds, with LLM generation 
representing 50% of total time (2.5s). Extract (0.2s) and reporting 
(0.2s) stages are negligible, while transformation (1.2s) and profiling 
(0.5s) constitute the remaining major components.
```

### Figure 2 - Memory Usage
```
Figure 2: Memory consumption profile over the ETL pipeline execution. 
Memory usage grows progressively from 50 MB at startup to a peak of 
200 MB during the transformation phase, with an average around 120 MB. 
This validates system feasibility even on resource-constrained machines.
```

### Figure 3 - Data Quality
```
Figure 3: Data quality improvement before and after cleaning across 
five application domains. The system improves quality by 29 points on 
average, ranging from +17% (financial data) to +38% (logistics data), 
demonstrating system adaptability to different data types and initial 
quality levels.
```

### Figure 4 - Throughput
```
Figure 4: Processing time and throughput analysis as a function of dataset 
size. The system maintains constant throughput of 20-26K rows per second 
with quasi-linear processing time. Left graph shows time evolution 
(1K to 1M rows), right graph illustrates throughput stability despite 
increasing data volume.
```

### Figure 5 - Anomaly Detection
```
Figure 5: Anomaly detection and correction rates by type on example 
dataset (1M rows). The system detects 30,320 total anomalies with 95.7% 
correction rate. NULL values constitute the major problem (12,500), 
followed by duplicates (5,600). Outliers are treated selectively (75%) 
to preserve potentially valid data.
```

### Figure 6 - LLM Accuracy
```
Figure 6: Precision of transformation rules generated by LLM across 
five evaluation metrics. Rules achieve 91% semantic similarity with 
manual rules, 86% transformation quality, and extremely low false 
positive rate (4%). Exact match (68%) indicates LLM often produces 
functionally equivalent variations rather than exact duplicates.
```

### Figure 7 - LLM Before/After RAG
```
Figure 7: Comparison of baseline LLM output quality versus RAG-augmented 
output across relevance, factuality, grounding, coverage, and coherence. 
The benchmark example shows an average gain from 65.2% to 90.2%, with the 
largest improvements coming from grounding and context coverage.
```

### Figure 8 - Cost Analysis
```
Figure 8: Comparative cost and duration analysis for three data cleaning 
approaches. Automated system reduces cost by 99.98% (from $200 to $0.03) 
and duration by 2,717x (from 240 minutes to 5.3 seconds) per dataset 
compared to manual cleaning, offering ROI justifying initial infrastructure 
investment.
```

### Figure 9 - Scalability
```
Figure 9: Comprehensive scalability test: processing time, memory 
consumption, and throughput. System demonstrates excellent scalability 
with quasi-O(n) time complexity, capable of efficiently processing 
datasets from 100 rows to 1 million rows. Memory grows linearly and 
remains predictable, while throughput stabilizes around 20K rows/second.
```

### Figure 10 - Transformations
```
Figure 10: Frequency and impact of eight transformation strategies applied. 
NULL filling (234x) and duplicate removal (245x) are dominant operations, 
jointly responsible for 85% of quality impact. Normalization (156x) and 
type conversion (12x) complete the ETL arsenal, comprehensively covering 
standard data cleaning categories.
```

### Figure 11 - Comparative Performance
```
Figure 11: Multi-criteria comparison of four data cleaning approaches. 
ETL+LLM system dominates on speed (9.5/10), cost (10/10), and 
maintainability (8/10) with composite score 36.1/40. Manual approaches 
only superior on accuracy (10/10 vs 8.6/10), representing acceptable 
compromise given massive gains in cost and speed.
```

## 🖼️ Display Recommendations

### For Presentations:
- Use 16:9 aspect ratio (already built in)
- 300 DPI supports projection without quality loss
- Test on actual projector before presentation
- Print slides at 100% for backup

### For Paper/Thesis:
- 300 DPI = professional print quality
- Black & white: OK (colors still differentiated)
- Single column: Use full width (0.9\textwidth)
- Double column: Reduce to 0.48\textwidth per column
- Ensure captions fit page width

### For Online/Website:
- PNG format: 72-96 DPI sufficient
- Use original 300 DPI files (users can zoom)
- Provide alt-text for accessibility
- Load images asynchronously for performance
